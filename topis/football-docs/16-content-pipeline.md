# CNT — 故事内容流水线与分发

> 承接你的判断：**本作的壁垒不是某一版内容，而是持续生产经过验证的
> 故事线种子的流水线**——玩家永远碰不到平淡内容，抄袭者抄到的只是
> 某个时刻的快照。本文件把这个判断机制化，并诚实评估服务端分发方案。

### CNT-01 故事线种子的定义与分层

状态: draft · 优先级: P0 · 依赖: EVT-07, DIR-03, SEED-05

**目的**：给「故事线种子」一个精确定义——它是内容更新的发货单位，
也是漏斗（15-mentor）验证的对象。

**设计理念**：种子不是剧本，是**经过模拟验证的戏剧原料组合**。
一个种子在不同世界里长出不同故事（G10），这是与「事件包 DLC」的
本质区别：卖的是会繁殖的基因，不是一次性的果实。

**如何设计**：
- 种子的四个层级（从轻到重）：
  S1 **事件簇**：3-8 个互相带 followup/记忆引用的事件定义 + 配套
  文本模板（EVT-01 schema 的集合，如「替补门将怨恨链」簇）；
  S2 **弧线模板**：DIR-06 的线头识别规则 + 阶段事件表（如「逼宫线」
  的完整定义）；
  S3 **人格/势力配方**：特质组合 + 老板/记者模板 + 文化标签的新搭配
  （ACT-01/WV-03/WV-04 的参数包）；
  S4 **世界配方**：SEED-05 生成参数包 + 戏剧种子注入表（SEED-02），
  决定「这一类世界」的整体气质；
- 每个种子的元数据：`{种子 id, 层级, 版本, 依赖的引擎能力版本,
  漏斗验证报告引用（15-mentor 的 L1/L3 结论）, 设计卡片来源 ID}`；
- 种子之间声明依赖与冲突（如两个事件簇共用同一记忆类型需对齐）；
- 命名约定进术语表：「故事线种子（story seed）」与「世界 seed
  （随机数 seed）」严格区分用词，文档与代码禁止混用。

**验收标准**：
- [机器] 种子包 schema 校验；依赖/冲突声明可解析且无环；
- [机器] 任意单个种子可独立通过试触发单测（EVT-01 同口径）；
- [机器] 种子元数据 100% 携带漏斗验证报告引用——**未经验证的种子
  不得发货**（流水线的质量承诺）。

**评估钩子**：
- 每种子入库后的实际触发率与玩家面曝光率（死种子清单）。

### CNT-02 内容包格式、版本化与确定性

状态: draft · 优先级: P0 · 依赖: CNT-01, SEED-03, CAT-06

**目的**：解决动态内容与确定性承诺（G5）的冲突——这是整个服务端
方案里**真正的技术难点**，比网络分发本身难得多。

**设计理念**：内容包是世界的一部分：同一存档必须永远用同一内容包
版本回放。鲜度属于新档，稳定属于旧档。

**如何设计**：
- 内容包 = 若干故事线种子的签名集合：`pack.json`（清单+语义版本+
  依赖的最低引擎版本）+ 种子数据 + 分发签名（Ed25519，客户端验签
  防篡改）；
- **存档钉住**：新档创建时记录 `{pack_id, pack_version, pack_hash}`
  入存档头（CAT-06 扩展）；读档时若本地无该版本包，从缓存/服务端
  取历史版本；取不到则明确告知「该存档需要内容包 vX」而非静默
  换包——同 seed 同包同版本 = 同世界线，缺一不可；
- seed 分享的语义升级（SEED-04 修订）：seed 名片 = 世界 seed +
  引擎版本 + 内容包版本三元组，社区讨论「同一个世界」时三者齐备；
- 包的兼容性政策：种子只增不改（修订=新种子 id + 弃用标记），
  使旧档在旧包上永远可复现；引擎能力升级才允许 breaking pack
  （与 SEED-03「打破世界线」政策同节奏，changelog 同步）；
- 基础包内置：游戏本体捆绑「base pack」（当前全部已验证种子），
  离线/无服务端时功能完整（CNT-04 的兜底前提）。

**验收标准**：
- [机器] 同（seed, 引擎版本, 包版本）三元组双跑世界线一致；
- [机器] 换包版本读旧档被拒绝并给出明确提示（不静默降级）；
- [机器] 包验签失败即拒载（防篡改路径可测）；
- [机器] 删除全部网络功能，base pack 单机全流程可玩。

**评估钩子**：
- 历史包版本的存档存活统计（多少玩家的档钉在多老的包上——
  决定历史包的保留周期）。

### CNT-03 持续生产流水线（壁垒的本体）

状态: draft · 优先级: P0 · 依赖: CNT-01, EVAL-05

**目的**：把「持续不断更新故事线种子」从愿望变成有节拍、有质检、
有度量的生产线——**这条流水线（评估语料库 + 规则库 + agent 工艺）
永不发货给客户端，才是真正抄不走的资产**。

**设计理念**：抄袭者能拿到的是种子快照；拿不到的是：六维评估语料库
（越用越准的代理评分，15-mentor §4）、蒸馏出的规则树（15-mentor §5）、
设计卡片库与 agent 工艺（studio）、以及你的迭代节拍。壁垒 =
流水线速度 × 验证数据积累，两者都随时间复利。

**如何设计**：
- 月度节拍（副业可持续版，对齐设计 agent 的 24 小时日循环）：
  W1 选题——评估报告弱项 + 玩家反馈 → 本月种子主题（如「老将
  退役线太薄」）；
  W2 批产——agent 按 EVT-07 规范起草种子草稿（事件簇/弧线模板）；
  W3 验证——种子注入测试包 → 夜批跑模拟 → 漏斗 L1 验收规则 +
  L3 LLM 品味抽检 → 不达标退回；
  W4 策展发布——人工终审（你的品味，唯一不可自动化的环节）→
  打包签名 → 推送 + 发布说明（「本月更衣室新增了什么暗流」本身
  是社区运营素材）；
- 产能目标：每月 1 个内容包 ≈ 1-2 个事件簇 + 若干模板/配方修订
  [待评估校准——第一个月实测产能后定]；
- 质量底线：任何种子入包前必须通过「与现有种子的组合跑批」
  （新旧种子互相引用/冲突的回归，不只测种子自身）；
- 度量：种子从选题到发布的周期、入包种子的玩家面曝光率、发包后
  评估六维分的版本趋势——流水线自己也有仪表盘。

**验收标准**：
- [机器] 流水线各环节产物有 schema、有归档（流程可审计）；
- [机器] 组合回归跑批纳入发包前置检查，不过不发；
- [人工] 连续 3 个月保持节拍且每月人工投入 ≤ 6 小时（副业可
  持续性的硬标准）。

**评估钩子**：
- 发包后 30 日的「内容疲劳指标」变化（EVT-05 重复感指标按版本
  对比——流水线存在的意义就是压住这条曲线）。

### CNT-04 服务端分发架构

状态: draft · 优先级: P1 · 依赖: CNT-02

**目的**：实现「每局游戏访问一次，拉取最新内容包」，同时不违背
单机游戏的离线承诺与个人运维承受力。直接回答：**这种做法难度低，
真正要花心思的是 CNT-02 的确定性治理和下面的离线兜底**。

**设计理念**：服务端做「报刊亭」不做「闸门」：在线时给鲜货，
离线时旧报照读。任何网络故障都不能让游戏不可玩——Steam 单机
玩家对强制在线的差评是毁灭性的，副业项目尤其赔不起口碑。

**如何设计**：
- 架构（个人运维最小化）：对象存储/CDN 托管静态签名包（无应用
  服务器、无数据库），客户端启动时一次 HTTPS 拉取
  `manifest.json` → 比对版本 → 后台下载新包入本地缓存 →
  **下一次新档**才启用新包（进行中的档继续钉旧版，CNT-02）；
- 成本与运维量级：静态托管月成本接近零（MB 级文件）；无服务端
  逻辑 = 无安全面 = 无值班——这是「难度低」的根据；
- 离线兜底链：本地缓存 → base pack（内置）→ 永远可玩；UI 仅
  温和提示「内容包 vX（最新 vY 可联网获取）」；
- 可选的轻遥测（同一次请求顺带）：匿名版本分布统计，用于决定
  历史包保留期（CNT-02 钩子）；不收集任何玩法数据 [玩法遥测
  是另一个决策，默认不做]；
- Steam 合规注意：商店页不标注「需要网络」；成就/存档全离线。

**验收标准**：
- [机器] 断网/弱网/服务端 404 三种情况下游戏完整可玩（自动化
  网络故障注入测试）；
- [机器] 包下载校验失败自动回退缓存版，无半成品包落地；
- [人工] 分发全链路（打包→签名→上传→客户端更新）一人 30 分钟
  内可完成（运维可持续）。

**评估钩子**：
- 客户端版本分布的收敛速度（发包后 30 日新包采用率）。

### CNT-05 防抄袭的现实评估与壁垒定位

状态: draft · 优先级: P1 · 依赖: CNT-03, CNT-04

**目的**：诚实回答「服务端放种子能否防恶意抄袭」，并把防抄袭
预算花在真有效的地方。

**设计理念**：凡是到达客户端内存的数据都可被导出——服务端分发
**提高抄袭成本但不构成技术壁垒**；真壁垒是 CNT-03 论证的「永不
发货的流水线资产」+ 更新节拍。防抄袭设计的目标因此修正为：
①防篡改（保护玩家体验一致性）；②抬高搬运成本与时滞；③让抄到的
快照随时间贬值。

**如何设计**：
- 有效手段（做）：
  包签名验签（CNT-02，防第三方塞私货冒充官方内容）；
  种子格式与引擎判定深度耦合（种子引用引擎的谓词/特质/记忆类型
  体系——搬运者必须连引擎语义一起仿，成本陡增）；
  更新节拍本身（抄袭者永远落后一个版本周期，且没有验证流水线，
  抄去的种子在他的引擎里大概率水土不服）；
  文本模板含风格指纹（轻量水印：特定措辞模式，维权取证用）；
- 不做的手段（性价比为负）：
  代码混淆/加密虚拟机（单机游戏破解史证明无效，且伤无辜玩家）；
  强制在线 DRM（口碑代价 > 抄袭损失，对副业体量尤其如此）；
  服务端算世界线（把模拟放服务端理论上防抄最彻底，但毁掉单机
  承诺、确定性回放、Mod 潜力与你的运维生活，明确否决）；
- 法务底线：种子内容文本登记著作权（中文文案是可确权资产）；
  开源边界明确（若引擎部分开源，种子内容与评估资产闭源）。

**验收标准**：
- [人工] 防抄袭措施清单逐项有「保护什么、代价是什么」的决策
  记录（本卡片即记录）；
- [机器] 签名/验签与风格指纹管线可测；
- [人工] 每半年复审一次威胁模型（真出现抄袭者时按实际形态调整，
  不预先过度设防）。

**评估钩子**：
- （上线后）社区搬运/仿制品监测记录与应对时滞。
