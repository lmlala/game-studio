# META — 设计卡片协议与全局原则

> 本文件是 football-docs 的「宪法」：定义卡片格式（供设计 agent 解析）、
> 全局设计原则（任何卡片不得违反）、统一术语表（防止 agent 改写时术语漂移）。

## 1. 卡片格式（机器可解析）

每张卡片是一个 `###` 级标题块，格式严格如下：

```markdown
### <ID> <中文标题>

状态: draft | refined | reviewed | locked · 优先级: P0 | P1 | P2 · 依赖: <ID 列表或「无」>

**目的**：一段话。这张卡片解决什么问题，不存在会怎样。

**设计理念**：一段话。为什么这样设计而不是别的，引用全局原则编号。

**如何设计**：
- 要点列表，写到「看完能开工」的细度；
- 涉及数值的给出初始建议值和允许区间；
- 涉及接口的给出输入/输出契约。

**验收标准**：
- 可逐条判定真/假的清单；能机器验证的优先机器验证。

**评估钩子**：
- 该设计接入 13-evaluation-pipeline 后用哪些指标/日志字段检验。
```

解析规则（设计 agent 必须遵守）：

1. ID 形如 `前缀-两位数字`（如 `DIR-04`），全库唯一，**永不复用、永不重排**；
2. 状态行必须是卡片标题后第一行非空行，使用全角间隔符 `·`；
3. 五个加粗字段名（目的/设计理念/如何设计/验收标准/评估钩子）**一个都不能少**，
   顺序不能变；agent 输出缺字段即判定为无效修订；
4. 卡片之间禁止隐式引用——提及其他卡片内容必须写 ID；
5. agent 只能修改 `draft` 与 `refined` 状态的卡片；`reviewed`/`locked` 只读；
6. 修订卡片时必须保留 ID 与标题，正文可重写；状态只能由人改为
   `reviewed/locked`，agent 最多把 `draft` 升为 `refined`。

## 2. 状态机

```text
draft ──(agent 自检通过：字段齐全、验收标准可判定、与依赖卡无冲突)──► refined
refined ──(人工 review 通过)──► reviewed ──(实现并验收)──► locked
任何状态 ──(评估数据证伪/人工打回)──► draft
```

## 3. 全局设计原则（编号供卡片引用）

- **G1 有限控制**：玩家只能做主教练能做的事；系统保留人心、媒体、老板的不确定性。
- **G2 足球规则不架空，足球世界架空**：规则层现实可信，内容层虚构。
- **G3 事件不是弹窗，是后果**：每个重大事件至少改变状态/关系/派系/记忆/联赛/
  比赛/老板压力/媒体压力之一。
- **G4 可解释性**：每个重大变化能回答谁触发、为何此刻、影响了谁、改了什么、
  写入了什么记忆。自然语言是展示层，结构化日志才是真值。
- **G5 确定性**：同 seed 同版本结果一致；存读档后继续与不中断一致。
- **G6 荒诞必须有因果锚点**：荒诞事件必须能从人物性格、利益、压力、误会、
  记忆或社会环境中找到原因（详见 DIR-04）。
- **G7 失败也要好看**：输球、降级、被解雇必须产出比胜利更密集的故事素材。
- **G8 个人副业可收敛**：任何设计先问「一个人业余时间能否做完最小版本」；
  宁可窄而深，不要宽而浅。
- **G9 Rust 权威，Godot 展示**：Godot 不持有权威状态；引擎可 CLI 无头运行。
- **G10 内容可组合**：事件、特质、记忆优先做可组合的积木，而不是一次性剧本。

## 4. 术语表（agent 改写时禁止替换为同义词）

| 术语 | 英文/代码名 | 定义 |
| --- | --- | --- |
| 引擎 | Engine | 负责**一种抽象判定**的纯逻辑模块，输入世界视图+RNG 流，输出判定/候选/日志 |
| 主叙事引擎 | Story Director | 从候选事件中按压力与风格发牌的调度器，不创造事实 |
| 事件 | Event | 结构化的世界变化记录：触发条件+效果+记忆写入+文本模板引用 |
| 候选事件 | Event Candidate | 子引擎提出、尚未被 Director 选中的事件 |
| 记忆 | Memory | Actor 持有的、带显著度与衰减的过去事件引用 |
| 关系 | Relationship | 有向二元情感向量（信任/尊重/亲密/嫉妒/怨恨…） |
| 派系 | Faction | 由关系图聚类产生的非固定团体，带议程与凝聚力 |
| 因果 ID | cause_id | 任何状态变化必须携带的来源事件/判定标识 |
| 荒诞度 | absurdity | 事件偏离现实常识的程度，0-3 级（见 DIR-04） |
| 压力曲线 | tension curve | Director 维护的目标戏剧张力轨迹 |
| 世界线 | world line | 多赛季尺度的世界演化轨迹（王朝、衰落、权力更替） |
| 战报 | season digest | 由结构化日志压缩生成的可分享赛季叙事摘要 |
| RNG 流 | RNG stream | 由主 seed 派生的命名随机数子流，按引擎/实体隔离 |
| 评估钩子 | eval hook | 卡片声明的、供跑批与 LLM 评估使用的指标定义 |

## 5. 卡片写作红线

1. 不写「尽量」「适当」「合理」这类不可判定词在验收标准里；
2. 数值必须给初始值+区间，没有把握就标注 `[待评估校准]`，由 EVAL 管线收敛；
3. 不发明新系统去解决可以用现有卡片组合解决的问题（G10）；
4. 单卡片「如何设计」超过 25 个要点时必须拆卡；
5. 验收标准必须区分「机器可验证」与「需 LLM/人工评估」，前者优先。
