# EVT — 事件系统

> 事件是世界变化的结构化记录单元（G3：事件不是弹窗，是后果）。
> 本文件定义事件的 schema、候选生成、效果应用、链式组合、防疲劳与文本渲染。

### EVT-01 事件定义 Schema

状态: draft · 优先级: P0 · 依赖: ENG-04, WV-05

**目的**：统一所有事件的数据结构，使事件可批量编写、机器校验、组合
引用，并成为设计 agent 可以批量生产/精修的内容单元。

**设计理念**：G10。事件定义是数据不是代码；新事件 = 新数据行。
schema 的表达力上限决定内容生产的天花板。

**如何设计**：
- 事件定义（静态，内容库）字段：

```text
id, category(冲突/温情/荒诞/政治/比赛/媒体/老板/球迷/伤病/转会/仪式),
absurdity(0-2), severity(minor/notable/major/season_defining),
prerequisites[](状态谓词列表: 阶段标签/关系阈值/特质持有/记忆存在/压力区间),
weight_base, weight_modifiers[](地区/文化标签/风格的乘数),
roles[](参与者槽位: 如 instigator/target/witness, 各带筛选谓词),
effects[](状态补丁模板: 目标域+delta范围),
memory_writes[](写给谁/记忆类型/显著度),
followup_hooks[](注册的后续钩子: 延迟天数+条件+目标事件id),
anchor_required(bool, absurdity≥2 必须 true),
cooldown_self(日), cooldown_category(日),
text_keys[](文本模板键, 按视角: 世界/球队/当事人),
tags[]
```

- 事件实例（动态，运行时）= 定义 id + 绑定的具体 Actor + 实际效果值 +
  cause_ids + tick；
- 谓词系统：前置条件用统一的谓词 DSL（字段·比较符·值 + and/or 树），
  机器可校验、设计 agent 可生成；
- 角色槽位筛选示例：「instigator = 同队 && 嫉妒(target) > 60 &&
  特质含 \"火爆\"」——槽位绑定失败则候选不成立。

**验收标准**：
- [机器] 全部事件定义通过 schema 校验；谓词引用的字段全部真实存在；
- [机器] 任意事件定义可被「试触发器」单测：构造满足前置的最小世界，
  事件能完整走完候选→选中→效果→记忆→日志；
- [机器] absurdity≥2 且 anchor_required=false 的定义数 = 0。

**评估钩子**：
- 事件定义的字段完备率、谓词复杂度分布（过简=触发烂大街，过繁=永不触发）。

### EVT-02 候选生成管线

状态: draft · 优先级: P0 · 依赖: EVT-01, ENG-02

**目的**：每 tick 从事件库 + 子引擎判定中产出本时刻「可能发生的事」，
供 Director 选牌。质量要求：候选既不能稀缺（Director 无牌可打）也不能
泛滥（评分失去意义）。

**设计理念**：两路候选源汇流：内容库驱动（事件定义的前置条件扫描）+
引擎驱动（子引擎在运行中主动提出，如训练冲突 CAT-01）。

**如何设计**：
- 扫描优化：事件定义按前置条件的主键（阶段标签/涉及状态域）建索引，
  只扫状态脏标记相关的定义，避免全库逐条求值；
- 候选结构：定义 id + 角色绑定 + 当前权重（base × modifiers × 引擎
  加成）+ 锚点引用（DIR-04 校验）；
- 候选量预算：每 tick 进入 Director 的候选 ≤ 30 [待评估校准]，超出按
  权重截断；
- 锚点硬校验在本层执行：absurdity≥2 无锚点直接丢弃并记 warning 日志
  （设计缺陷信号，进评估报告）；
- 玩家命令产生的「直接事件」（处罚、表扬、发布会）不走候选，在阶段 1
  校验后阶段 9 直接应用——玩家行为必然发生，后果才走候选。

**验收标准**：
- [机器] 跑批中每 tick 候选数分布 P95 ≤ 30、P5 ≥ 2（无饥荒无洪水）；
- [机器] 角色槽位绑定零违例（绑定的 Actor 全部满足槽位谓词）；
- [机器] 候选生成耗时 < 单 tick 预算的 30%（ENG-06）。

**评估钩子**：
- 候选→选中转化率按类别统计（某类长期 0 选中 = 权重或内容问题）。

### EVT-03 效果应用与事务性

状态: draft · 优先级: P0 · 依赖: EVT-01, ENG-01

**目的**：保证事件效果原子、有序、可追溯地写入世界状态，避免半应用
状态与跨事件干扰。

**设计理念**：阶段 9 是全 tick 唯一的「世界写入窗口」（玩家直接命令
除外），所有补丁排序后确定性应用（G5）。

**如何设计**：
- 应用顺序：先玩家直接事件，再 Director 已选事件按选牌序；同一目标
  字段多补丁按「先到先应用、后者基于新值」处理；
- 范围效果展开：效果模板的 delta 范围（如 心情 -5~-15）在应用时用
  事件专属 RNG 流定值并记录实际值（before/after/delta 进日志）；
- 级联上限：事件效果可触发的即时连锁（如关系破裂触发派系重算）深度
  ≤ 2，更深的涟漪交给后续 tick 的引擎自然消化——防止单 tick 雪崩；
- 失败处理：补丁目标不存在（Actor 已转会等）则该补丁静默丢弃并记
  warning（不是 panic：候选生成与应用之间可能隔一个比赛阶段）。

**验收标准**：
- [机器] 同 seed 下补丁应用序逐字节稳定；
- [机器] 每条状态变化日志的 before+delta=after 恒等式校验通过；
- [机器] 构造 100 个并发补丁的压力单测，无丢失无重复应用。

**评估钩子**：
- warning 日志率（持续上升说明事件定义与世界状态脱节）。

### EVT-04 事件链与记忆引用

状态: draft · 优先级: P0 · 依赖: EVT-01, ACT-03, DIR-06

**目的**：把孤立事件连成有延迟、有回响的链条——「半年后年轻球员开始
觉得我会牺牲球员」这类 MVP 成功句式的机制载体。

**设计理念**：链 = followup 钩子（确定的延迟引信）+ 记忆引用（概率的
旧事重提）。前者保证关键剧情不断线，后者制造惊喜回响。

**如何设计**：
- followup 钩子：事件应用时注册 `{延迟区间, 触发条件谓词, 目标事件 id,
  携带的 cause_id}`，到期且条件满足则目标事件直接进候选并带继承权重
  ×2；条件不满足则过期作废并写「不了了之」微日志（线头有交代）；
- 记忆引用事件：一类特殊事件定义，前置条件含「存在记忆 M 且显著度 >
  阈值」，触发时文本与效果引用该记忆内容（如旧承诺被违背→引用当时
  的承诺记忆）；
- 链深统计：cause_id DAG 上的最长链长度是核心叙事健康指标——链太短
  = 世界没有记性，太长 = 可能逻辑失控；目标带 3-8 [待评估校准]；
- MVP 内容要求：事件池中 ≥ 30% 的定义具有 followup 钩子或记忆引用
  前置（EVT-07 的写作规范）。

**验收标准**：
- [机器] followup 钩子 100% 有终态（触发/过期），无悬挂；
- [机器] 记忆引用事件触发时引用的 memory_id 真实存在且属于当事人；
- [LLM] 抽 10 条 3+ 长度事件链，「前后呼应感」评分 ≥ 4/5。

**评估钩子**：
- cause 链长度分布；记忆引用事件占已发生事件比例（目标 ≥ 15%
  [待评估校准]）。

### EVT-05 冷却、去重与反疲劳

状态: draft · 优先级: P1 · 依赖: EVT-02, DIR-01

**目的**：对抗事件池疲劳——商业风险表「内容量不足，事件重复很快疲劳」
的机制侧防线（内容侧防线是 EVT-07）。

**设计理念**：重复不可避免，可控的是重复的「间隔、换装、视角」。
同一定义再次出现时必须至少换参与者或换语境。

**如何设计**：
- 三级冷却：定义级（同 id 冷却 N 日）、类别级（同 category 软冷却，
  权重衰减而非禁止）、参与者级（同一对 Actor 的同类冲突 30 日内
  权重 ×0.3）；
- 多样性奖励：Director 评分中「近 30 日未出现的类别」候选 ×1.3
  （DIR-01 公式的多样性项）；
- 换装机制：同一定义绑定多套文本模板（text_keys ≥ 2 套），重复触发
  时强制轮换；
- 疲劳监测指标直出评估管线：滑动窗口内定义重复率、文本模板重复率。

**验收标准**：
- [机器] 单赛季同一事件定义触发 ≥ 3 次的比例 < 15% [待评估校准]；
- [机器] 同定义连续两次触发使用不同文本模板；
- [LLM] 评估器整季战报「事件重复感」报告率 < 15%。

**评估钩子**：
- 事件定义使用熵（全池利用率）；零触发定义清单（死内容，退给设计）。

### EVT-06 文本渲染层（模板 + LLM 扩写）

状态: draft · 优先级: P1 · 依赖: EVT-01, ENG-04

**目的**：把结构化事件渲染成中文（后英文）展示文本。这是你写作短板的
系统性补偿：人写模板骨架，LLM 做风格化扩写，结构化数据永远是真值。

**设计理念**：三层渲染金字塔——80% 走纯模板（零成本零延迟）、15% 走
模板+插槽变体、5%（major/season_defining）可选走 LLM 扩写。LLM 输出
仅是展示缓存，删除后游戏照常可玩（G4：自然语言是显示数据）。

**如何设计**：
- 模板语法：message_key → 带槽位的文本（人名/队名/数值/记忆摘要槽），
  支持按视角（世界新闻/球队内部/当事人独白）和语气键（DIR-03 风格）
  选不同模板组；
- LLM 扩写管线（可离线/可关闭）：输入 = 事件实例 + 参与者人设卡 +
  相关记忆摘要 + 风格语气键；输出 ≤ 120 字中文段落；缓存进存档侧表；
- 一致性约束：LLM 输出做后校验——出现未在输入中的实体名/数值即丢弃
  回退模板（防幻觉污染叙事真值）；
- 离线包模式：无网/无 key 时全模板运行，功能零缺失（商店页承诺单机
  可玩的底线）。

**验收标准**：
- [机器] 全部 message_key 有 zh 模板；模板槽位与事件 schema 字段对齐；
- [机器] LLM 后校验拒绝率 < 20%（过高说明 prompt 或模板分工有问题）；
- [机器] 关闭 LLM 跑完整赛季，战报完整生成；
- [LLM] 同一事件模板版 vs 扩写版做偏好盲测，扩写版胜率 ≥ 60% 才保留
  该类扩写（否则纯模板，省成本）。

**评估钩子**：
- 文本重复率（n-gram）；扩写缓存命中率与成本统计。

### EVT-07 事件池内容规划与写作规范

状态: draft · 优先级: P0 · 依赖: EVT-01, WV-05, 全部 CAT 卡片

**目的**：规定 MVP 事件池的规模、配比与写作质量标准——设计 agent
批量生产事件定义时的验收依据。

**设计理念**：宁要 80 个可组合、带钩子、多视角的事件，不要 300 个
一次性弹窗。每个事件是积木不是剧本（G10）。

**如何设计**：
- MVP 规模：80-120 个定义；类别配比（初始，[待评估校准]）：
  冲突 20% / 比赛余波 15% / 媒体 12% / 老板 10% / 球迷 8% / 温情和解
  10% / 伤病身体 8% / 荒诞 8% / 转会风闻 5% / 仪式节点 4%；
- 结构性要求（每个定义逐项检查）：
  1. 至少 1 个效果触达 G3 列表（状态/关系/派系/记忆/联赛/比赛/老板压力/媒体压力）；
  2. ≥ 30% 定义带 followup 或记忆引用（EVT-04）；
  3. 每定义 ≥ 2 套文本模板（EVT-05）；
  4. 文化标签专属事件每标签 ≥ 5 个（WV-03）；
  5. 失败/危机类合计 ≥ 25%（DIR-08）；
- 写作风格规范：标题 ≤ 20 字、正文模板 ≤ 80 字、禁止解释机制术语、
  黑色幽默来自利益错位而非段子（WV-04 理念）、当事人视角必须体现其
  人格特质用词；
- 生产流程：设计 agent 按本规范批量起草 → 谓词机器校验 → 试触发单测
  → 人工抽检 20% → 入库。

**验收标准**：
- [机器] 池规模与配比在目标区间；5 项结构性要求全部通过脚本检查；
- [机器] 全部定义通过试触发单测（EVT-01）；
- [LLM] 入库前每定义过「人设一致性+幽默感+无机制泄漏」三项评审，
  均分 ≥ 3.5/5 方可入库。

**评估钩子**：
- 跑批后的零触发定义率 < 10%；类别实际触发配比 vs 设计配比偏差。
