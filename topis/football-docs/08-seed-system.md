# SEED — seed 系统：确定性、RNG 流、世界生成

> seed 系统是整个项目的工程信用：同 seed 同世界（可分享、可复现、可
> 回归测试），也是评估管线（EVAL）与设计 agent 闭环的前提（G5）。

### SEED-01 RNG 流架构

状态: draft · 优先级: P0 · 依赖: 无

**目的**：把「一个 seed」变成「一棵确定性随机树」，使任何引擎、任何
实体的随机消费互相隔离——新增引擎或事件不扰动其他系统的随机序列。

**设计理念**：随机数是状态不是服务。全局共享一个 RNG 是确定性的头号
杀手（消费顺序一变全盘漂移）；命名子流让随机性模块化。

**如何设计**：
- 派生树：`master_seed → domain_seed = H(master, domain_name) →
  stream = H(domain, entity_id, epoch)`；H 用 SplitMix64/xxhash 类
  确定性混合；PRNG 用 PCG64 或 xoshiro256**（可序列化）；
- 域划分（与引擎对应，ENG-03）：worldgen / match.{match_id} /
  training.{club_id} / social.{club_id} / event_roll / director /
  effect.{event_instance_id}…；每域每纪元（赛季）换 epoch 防长序列
  相关性；
- 消费纪律：引擎只能用注入的本域流（trait 签名强制，ENG-03）；
  禁止 `thread_rng()`、禁止用时间/地址做熵源（lint 规则）；
- 全部流状态入存档（CAT-06）；读档后续跑 = 不中断（G5 的机制保证）；
- 浮点纪律：跨平台一致性要求——禁用快速数学优化，比较运算定阈值，
  累加固定顺序 [若日后跨平台漂移，升级为定点数策略，待评估]。

**验收标准**：
- [机器] 同 seed 双跑全 tick 状态哈希一致；增加一个新引擎后，其余
  引擎的随机序列逐值不变（隔离性测试）；
- [机器] lint：源码零 `thread_rng`/系统时间熵源；
- [机器] 存读档点随机插入 100 次，续跑与对照完全一致。

**评估钩子**：
- 跑批前置自检：每次批跑首尾各做一次确定性双跑校验，失败即中止
  （脏数据不进评估）。

### SEED-02 世界生成管线

状态: draft · 优先级: P0 · 依赖: SEED-01, WV-01, WV-03, ACT-01, REL-01

**目的**：从一个 seed 生成完整可玩世界：国家→俱乐部→人物→初始
关系网→赛程。生成质量决定前 2 小时体验（情感冷启动的供给侧）。

**设计理念**：生成不是均匀抽样，是**带戏剧潜能的布局**：开局世界
必须自带张力（强弱差、恩怨对、派系雏形、不稳定因素），玩家进场时
火药桶已经摆好。

**如何设计**：
- 流水线（固定顺序，各阶段独立子流）：
  1. 国家与地区参数实例化（WV-01）；
  2. 俱乐部生成：文化标签组合（互斥规避）、财政档位、历史底蕴值、
     预置 2-3 对德比/恩怨关系（WV-03）；
  3. 联赛强弱布局：实力分层抽样（2 强 / 4-6 中 / 2-4 弱），保证
     竞争格局而非平地；
  4. 人物生成：按队按位置配额生成球员（年龄/能力/人格联合分布，
     地区调制），再生成老板/记者/助教等（ACT-07）；
  5. 戏剧种子注入：每队按文化标签注入 2-4 个不稳定因素（顶薪老将+
     天才新秀同位置 / 合同年核心 / 旧伤隐患 / 跟前任教练的旧怨记忆）
     ——这是「第一周就有事可讲」的保证；
  6. 初始关系网与潜在派系雏形（REL-01/03）；
  7. 赛程生成（WV-02）；
- 玩家球队选择：生成后展示每队一句话处境摘要（带剧透度控制），
  默认推荐 1 支「中游多事之队」（叙事密度最高的新手村）；
- 生成报告：每个新世界产出结构化「世界出厂报告」（强弱表/恩怨表/
  不稳定因素表），供调试与评估（EVAL-02 的生成质量指标输入）。

**验收标准**：
- [机器] 同 seed 生成世界逐字节一致；生成耗时 < 3s；
- [机器] sanity 检查全过：位置配额、年龄结构、能力分布、标签互斥、
  名字查重（WV-06）；
- [机器] 每队不稳定因素 2-4 个且类型不全同；
- [LLM] 抽 3 队出厂报告评「开局处境是否让人想接手」≥ 3.5/5。

**评估钩子**：
- 100 seed 生成统计：能力分布方差、恩怨对数量、潜在派系数——
  生成多样性指标（防止千篇一律的开局）。

### SEED-03 确定性契约与回归测试

状态: draft · 优先级: P0 · 依赖: SEED-01, ENG-02

**目的**：把 G5 从口号变成持续执行的测试合同——设计 agent 改配置、
你改代码之后，机器第一时间发现确定性破坏。

**设计理念**：确定性是会腐烂的属性（一个 HashMap 迭代就毁掉）；
必须用 CI 级别的自动合同看住，而不是靠记忆力。

**如何设计**：
- 契约条款（全部为自动测试）：
  1. 双跑一致：同 seed 同版本，全 tick 状态哈希序列相等；
  2. 存读等价：任意 tick 存读后续跑 = 不中断；
  3. 隔离性：注册空白引擎后其他引擎随机序列不变（SEED-01）；
  4. 迭代序稳定：全部集合遍历用 BTreeMap/排序 Vec（lint + 抽查）；
  5. 配置等价：仅改展示层配置（文本/UI）不改状态哈希；
- 版本变更政策：游戏逻辑变更**允许**打破旧 seed 的世界线（写入
  changelog），但同版本内永远稳定；存档带版本号拒绝跨版本盲读
  （CAT-06）；
- 黄金档（golden saves）：维护 3-5 个固定 seed 的标志性存档+预期
  指标快照，重大重构后人工对照「世界还认识吗」。

**验收标准**：
- [机器] 上述 5 条契约测试全部进 CI，任一失败阻断合并；
- [机器] lint 规则（禁 HashMap 迭代入逻辑、禁 thread_rng）零违例；
- [人工] changelog 中每次「打破世界线」的变更有记录与理由。

**评估钩子**：
- CI 确定性测试历史通过率；黄金档指标漂移记录。

### SEED-04 seed 的产品化（可分享世界）

状态: draft · 优先级: P2 · 依赖: SEED-02, SEED-03

**目的**：把 seed 从工程工具升级为传播货币：「试试 seed 8742，第三周
更衣室就炸了」——零成本的社区内容（用户画像四：战报玩家）。

**设计理念**：RimWorld/矮人要塞社区证明：可分享的世界种子 + 可复述
的故事 = 自传播。副业项目营销预算为零，产品自带传播性是生死项。

**如何设计**：
- seed 输入框新档可见（已有约定保留），seed 显示在战报头部与截图
  水印位；
- seed 卡片导出：一键生成「世界名片」图（seed + 联赛格局图 + 一句
  最荒诞开局事实），尺寸适配社交媒体（UX-05 联动）；
- 名场面坐标：seed + 赛季 + 周数可精确定位事件，社区可以讨论「同一
  个世界你玩到第 10 周发生了什么」（同 seed 不同玩家决策的分叉对比
  ——本作独有的传播形态）；
- 版本绑定提示：seed 卡片带版本号，跨版本明示「世界线已变」（SEED-03
  政策的玩家面表达）。

**验收标准**：
- [机器] seed 卡片导出功能可用，含全部要素；
- [机器] 同 seed 不同玩家命令序列，分叉前世界一致（部分确定性验证）；
- [人工] 内测期社区出现 ≥ 1 次自发的 seed 分享讨论（真实验证）。

**评估钩子**：
- （上线后）含 seed 的社区帖计数——传播性假设的直接检验。

### SEED-05 生成参数的可调面板

状态: draft · 优先级: P2 · 依赖: SEED-02

**目的**：把世界生成的关键分布参数暴露为数据文件，供评估管线扫参
（「哪种世界配方叙事密度最高」）与后期玩家自定义（混乱度滑杆）。

**设计理念**：生成参数是叙事质量的隐藏旋钮；让 agent/评估能系统性
探索参数空间，而不是手感调参（设计 agent 闭环的素材）。

**如何设计**：
- 参数文件（RON/JSON）：强弱分层比例、人格分布均值方差、不稳定
  因素配额、初始恩怨密度、记者偏见强度等 15-25 个旋钮；
- 每参数带元数据：取值域、默认值、影响的评估指标列表（EVAL-02
  对应项）——agent 调参时按图索骥；
- 玩家面（后期）：3 档预设（平稳/标准/火药桶）映射到参数包，不暴露
  原始旋钮（G8，不做参数沼泽 UI）。

**验收标准**：
- [机器] 参数文件 schema 校验、全部旋钮被生成器真实消费；
- [机器] 扫参跑批脚本可用：给定参数网格自动批跑并汇总指标对比。

**评估钩子**：
- 参数敏感性报告：每旋钮对核心叙事指标的弹性（EVAL-04 产出）。
