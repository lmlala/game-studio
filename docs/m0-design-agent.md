# AGT — 自写设计 agent 方案

> 直接回答：**自己写，合理，而且是正确选择**——前提是把它写成
> 「确定性流水线 + LLM 纯文本函数」，而不是 Claude Code 式的自主
> 工具调用循环。设计精修任务不需要工具调用，这正是中档模型
> （DeepSeek / Qwen）能胜任的原因。
>
> **演进说明**：本文件定义的单角色闭环 = 完整框架的 **M0 里程碑**。
> 多角色多轮迭代、模拟用户、tool/skill、跨项目复用与交接包的完整
> 开发设计见 [`framework/`](framework/README.md)；
> 本文件的流水线、门禁、上下文组装、模型策略全部被框架继承。
>
> 位置说明：本文件与 framework/ 是 **design-studio 的软件设计文档**，
> 不属于游戏设计，故从 football-docs 迁至此处，随应用一起迁出仓库。

## 为什么这个场景适合自写

| 对比项 | 编码 agent（Claude Code 等） | 本设计 agent |
| --- | --- | --- |
| 任务 | 改代码、跑命令、看报错 | 改 markdown 卡片 |
| 核心难点 | 工具调用循环、编辑可靠性、环境状态 | 上下文组装、一致性校验 |
| 自主性需求 | 高（模型决定下一步） | 零（流程由你的代码决定） |
| 模型要求 | 前沿模型 | 中档模型 + 量表锚定 |
| 自写成本 | 数周（不值得） | 300-600 行 Python（值得） |

Claude Code 的「简单 loop」简单在概念，贵在工程细节（diff 应用、
沙箱、上下文压缩）。而设计卡片精修可以**绕过全部这些难点**：
你的编排代码决定读哪些文件、写哪个卡片，模型只做「文本进、文本出」
——没有工具调用就没有工具调用的坑。

### AGT-01 总体决策与边界

状态: draft · 优先级: P0 · 依赖: 00 卡片协议

**目的**：锁定设计 agent 的职责边界，防止范围蔓延成自研编码 agent。

**设计理念**：agent 是「不知疲倦的设计助理编辑」，不是设计师：
它批量做你没耐心做的事（细化、查漏、对齐、按评估证据修订），
品味决策永远在你（人工闸门 EVAL-05）。

**如何设计**：
- 职责内：精修 draft/refined 卡片、按任务卡（EVAL-05）修订、
  批量起草事件定义（EVT-07 规范下）、卡片间一致性巡检、为新引擎
  起草卡片初稿；
- 职责外（明确禁止）：改代码、改 `docs/`（旧权威文档）、改
  reviewed/locked 卡片、改 00 协议文件本身、发明新卡片 ID 段
  （新 ID 由你分配）；
- 运行形态：本地 CLI（`python design_agent.py run --max-cards 10`），
  cron 夜跑或手动触发；输出全部走 git 分支 + diff，你早上 review；
- 升级路径：等设计稳定进入实现期，编码任务交给现成编码 agent
  （Claude Code / Codex CLI 按需购买），两套各管各——不要让设计
  agent 长出手脚。

**验收标准**：
- [机器] agent 对禁区文件的写操作被代码级拦截（不是靠 prompt 约束）；
- [机器] 全部修改可通过 `git diff` 审计，无绕过 git 的写入；
- [人工] 运行 2 周后审视：agent 产出中你采纳率 ≥ 60%（低于此值
  说明 prompt/量表需要返工，而不是换大模型先）。

**评估钩子**：
- 每卡片修订的人工采纳率（agent 的 KPI，按周追踪）。

### AGT-02 流水线架构（确定性编排 + LLM 纯函数）

状态: draft · 优先级: P0 · 依赖: AGT-01

**目的**：定义 agent 的执行模型——你问的「简单 loop 是否可以」：
可以，但要是**编排在你手里的 loop**，不是模型自主的 loop。

**设计理念**：模型的每次调用都是无状态纯函数（prompt 进、JSON 出）；
状态、流程、文件 IO 全在 Python 编排层。可测、可断点、可换模型。

**如何设计**：
- 五段流水线（每张卡片一轮）：

```text
1 SELECT   选卡: 任务队列(_tasks/) 优先, 否则按 状态=draft × 优先级 × 依赖已就绪 排序
2 ASSEMBLE 组装上下文(AGT-04): 卡片全文 + 依赖卡摘要 + 00协议 + 相关评估证据
3 CRITIQUE 评审调用: 按检查清单输出 JSON {违例[], 缺陷[], 各项评分, 是否需要重写}
4 REWRITE  重写调用(仅当需要): 输出完整新卡片文本; 不需要则只做状态升级判定
5 VALIDATE 机器校验(AGT-05): schema/术语/引用/字段完备 → 通过则写文件+git commit,
           失败则带错误信息重试 1 次, 再失败丢弃并记录
```

- 关键实现选择：
  无对话历史——每轮上下文从文件系统重新组装（卡片即记忆，
  天然持久化，重启零损失）；
  无函数调用 API 依赖——只用 chat completion + JSON mode，
  DeepSeek/Qwen/任何 OpenAI 兼容端点即插即用；
  并发按卡片粒度（不同卡片可并行，同文件卡片串行）；
- 技术栈：Python 3.12 + `openai` SDK（兼容端点）+ `pydantic`
  （卡片与 JSON 输出的 schema 校验）+ `gitpython` 或 subprocess
  git。**不用 LangChain/agent 框架**——本流水线总代码量比学习
  任何框架都少；
- 规模预估：编排 ~200 行、校验器 ~150 行、prompt 模板 3 个
  ~100 行、CLI/配置 ~100 行，合计 < 600 行（含注释）。

**验收标准**：
- [机器] 流水线对同一卡片同一上下文可重放（温度 0 时输出稳定）；
- [机器] 中断恢复：任意阶段 kill 进程，重启不产生半写卡片
  （写文件原子化：临时文件 + rename）；
- [机器] 换模型只改配置文件中的 endpoint/model 字段。

**评估钩子**：
- 每轮各阶段耗时与 token 消耗（瓶颈定位）。

### AGT-03 模型策略（DeepSeek / Qwen 中档模型）

状态: draft · 优先级: P0 · 依赖: AGT-02

**目的**：回答「用中档模型减少开销」是否成立：成立，且有具体的
扬长避短打法。

**设计理念**：中档模型的短板是开放性品味与长程规划，长板是按明确
量表执行结构化任务。本流水线的全部设计（锚定量表、检查清单、
JSON 输出、窄上下文）都在把任务从「品味题」改写成「执行题」。

**如何设计**：
- 分工矩阵：
  CRITIQUE/REWRITE 批量轮——DeepSeek V4 / Qwen 旗舰档（兼容
  API，温度 0.2 以下）；
  每周一次「主编轮」——强模型（Claude/GPT 当期旗舰）只评审本周
  全部已通过卡片的**摘要**（不重写，只标记「这张有问题」打回）
  ——强模型当质检不当工人，用量极小；
  事件文案起草（EVT-07 批产）——中档模型即可，文案质量由 EVT-07
  的三项 LLM 评审兜底（评审也用中档+抽检）；
- 成本估算（量级判断）：单卡片一轮 ≈ 上下文 8k + 输出 2k token；
  每晚 20 卡 × 30 天 ≈ 6M token/月，按 DeepSeek 档定价 ≈
  ¥10-30/月——**设计阶段的 API 成本可以忽略不计**，真正的成本
  在 EVAL-03 的评估层（已有预算卡 EVAL-06）；
- 模型评测脚本：固定 10 张测试卡 + 人工标准答案要点，新模型/
  新版本先跑评测再上岗（换模型不靠感觉）；
- 输出纪律：全部调用带 JSON schema 约束与重试；拒绝输出自由格式。

**验收标准**：
- [机器] 中档模型在测试卡集上的 CRITIQUE 违例检出率 ≥ 80%
  （以人工标注为准）；
- [机器] 月度 token 账单在估算 ±50% 内；
- [人工] 主编轮打回的卡片中，你认同打回理由的 ≥ 70%（强模型
  质检有效性）。

**评估钩子**：
- 采纳率按模型分组对比（中档够不够，数据说话，不预设结论）。

### AGT-04 上下文组装规则

状态: draft · 优先级: P0 · 依赖: AGT-02, 00 卡片协议

**目的**：中档模型成败的第一变量是喂什么。规则化上下文组装，
保证每次调用「该知道的都在、不该塞的不塞」。

**设计理念**：上下文是配餐不是自助餐：固定菜单 + 按卡片类型微调，
总量封顶（中档模型长上下文衰减明显，宁可窄而准）。

**如何设计**：
- 固定菜单（按顺序）：
  ⓪ `00a-game-overview.md` 游戏总览（≤2k token，"游戏是什么"的
  常驻锚点，长期不变，享受 prompt 缓存）；
  ① 00 协议的卡片格式节 + 全局原则 + 术语表（≈1.5k token，
  长期不变，享受 prompt 缓存）；
  ② 目标卡片全文；
  ③ 依赖卡片的「目的+如何设计」节选（每张 ≤ 400 token，超长
  机器截断）；
  ④ 同文件相邻卡片的标题+目的（防止卡片间内容重复/冲突）；
  ⑤ 相关评估证据（仅任务卡驱动时：指标值+最差时刻样例，
  EVAL-05）；
  ⑥ 该卡片的历史修订摘要（最近 2 次的 diff 概要——防止改回
  老方案来回拉锯）；
- 总量封顶 12k token [待评估校准]；超限按 ⑥→④→③ 顺序裁剪；
- 摘要预生成：依赖卡摘要在卡片提交时离线生成缓存（`_cache/`），
  避免每轮重复摘要调用。

**验收标准**：
- [机器] 上下文组装确定性（同卡片同库状态组装结果一致）；
- [机器] 超限裁剪逻辑单测；缓存失效正确（卡片改了摘要重生成）；
- [人工] 抽查 10 次调用的上下文，无「缺关键依赖」与「塞无关
  内容」案例。

**评估钩子**：
- 上下文长度分布；裁剪发生率（持续高位 = 卡片写太长或依赖过密，
  退给 00 协议的拆卡规则）。

### AGT-05 质量门禁与防退化

状态: draft · 优先级: P0 · 依赖: AGT-02, 00 卡片协议

**目的**：agent 批量修改的最大风险不是写得差，是**悄悄变差**
（术语漂移、验收标准变空话、卡片间矛盾累积）。门禁让退化在
合入前被机器拦截。

**设计理念**：信任模型的产出，验证模型的产出——全部校验在
VALIDATE 阶段用确定性代码做，不用「再问一次模型对不对」
（中档模型自我评审不可靠）。

**如何设计**：
- 机器门禁清单（全部正则/解析级实现，零 LLM）：
  schema——五字段齐全、状态行合法、ID 未被篡改；
  术语 lint——00 术语表词条的同义词替换检测（如「主叙事引擎」
  被写成「故事调度器」即拦截）；
  引用检查——文中出现的全部卡片 ID 真实存在；依赖声明与正文
  引用一致；
  可判定性 lint——验收标准节中的禁用词扫描（「尽量/适当/合理/
  良好」出现即打回，00 红线 1）；
  膨胀检测——卡片长度增幅 > 50% 时要求 CRITIQUE 阶段曾给出
  扩写理由，否则打回；
- 振荡检测：卡片连续 3 次修订的 diff 相似度过高（来回改同一段）
  → 冻结标记 `needs-human`（EVAL-05 同款）；
- 每日巡检模式（与精修分开的轻量轮）：全库跑机器门禁 + 跨文件
  重复内容检测（n-gram），产出巡检报告不改文件；
- git 纪律：每卡片一 commit（message 含卡片 ID 与修订理由摘要），
  每日一分支一 PR——你 review 的单位是 PR，回滚的单位是 commit。

**验收标准**：
- [机器] 门禁全项有单测；故意构造的 10 种坏修订全部被拦；
- [机器] 振荡冻结触发后该卡片不再被 SELECT；
- [人工] 运行一个月后人工全库抽查 10%：未发现门禁漏放的术语
  漂移/空话验收标准。

**评估钩子**：
- 门禁拦截率分布（哪类错误最多 = prompt 改进方向）；冻结卡片
  数量趋势。

### AGT-06 工作流：副业节奏的 24 小时循环

状态: draft · 优先级: P1 · 依赖: AGT-01..05, EVAL-05

**目的**：把 agent、评估、你三方排进一个可持续的业余时间循环——
工具为节奏服务，节奏为坚持服务（G8 的工程化）。

**设计理念**：你的不可替代时间只花在三件事：批任务、review diff、
做品味决策。其余一切自动。

**如何设计**：
- 日循环：
  夜间（cron）——评估跑批（若有代码/参数变更）→ EVAL-05 生成
  任务卡 → agent 精修轮（max-cards 限额）→ 推送分支开 PR；
  早晨（你，≤ 15 分钟）——批任务队列 + 扫 PR diff（手机可完成，
  diff 是 markdown 天然可读）；
  晚上（你，正餐时间）——合并/打回 PR、做被冻结卡片的人工决策、
  改 prompt/门禁（agent 的成长来自这里，不是模型变大）；
- 周循环：主编轮（AGT-03 强模型质检）→ 周报（卡片状态统计 +
  采纳率 + 评估趋势线一张图）；
- 状态推进节流：每周进入 reviewed 的卡片 ≤ 10 张——你的 review
  带宽是稀缺资源，宁可 agent 等你，不可你追 agent；
- 全部脚本进 `tools/design_agent/`（与现有 tools/ 并列），README
  含一键安装与 cron 配置示例。

**验收标准**：
- [机器] 全循环无人值守可运行（任一环节失败不阻塞次日，带告警）；
- [人工] 连续 4 周保持循环后自评：每周人工投入 ≤ 3 小时且卡片
  库可感知地变好（最终标准就是这一条）。

**评估钩子**：
- 周吞吐（精修卡片数/周）、你的实际投入时间记录、采纳率趋势
  ——三线合看判断这套系统是否值得继续。
